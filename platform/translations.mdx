The WCA platform supports over 30 locales, making it accessible to the global cubing community in their native language. Translations cover both the server-rendered Rails views and the React frontend components.

<Note>
  Translation completeness varies by language. The translations status page at `/translations/status` shows which locale files have missing or outdated keys compared to the English source.
</Note>

## How translations work

The WCA platform uses two translation systems working in tandem:

**Rails i18n** handles all server-rendered content. Locale files live in `config/locales/` as YAML files, one per locale (e.g., `config/locales/fr.yml`). Keys are nested under the locale code at the top level:

```yaml
fr:
  competitions:
    index:
      title: "Compétitions"
  common:
    save: "Enregistrer"
```

**i18n-js** exports a subset of the Rails locale files into JSON files consumed by the React frontend. The export is configured in `config/i18n.yml`, which specifies which key patterns to include. The exported files are written to `app/webpacker/rails_translations/:locale.json`. Only keys matching the listed patterns are included — for example, `*.competitions.*`, `*.common.*`, `*.events.*`, and so on.

When a user visits the site, their browser locale preference (or their account's saved language setting) determines which locale is active. The selected locale applies to both server-rendered pages and React components simultaneously.

## Translation status page

Visit `/translations/status` to see the current state of all locale files. The page is generated by `TranslationsController#compute_bad_i18n_keys`, which compares each locale's YAML file against its base locale (English for most, French for `fr-CA`).

The comparison identifies three categories of problems:

- **Missing keys:** A key present in the base locale file is absent from the locale file.
- **Extra keys:** A key is in the locale file but not in the base locale — usually left over from a previous version of the site.
- **Outdated translations:** The value matches the base locale exactly, suggesting it was never translated (or was accidentally reset).

If all locales are complete, the page shows a success message. Otherwise, each locale's issues are listed with the affected key paths.

## Contributing a translation

You can submit a translation update directly through the WCA website. The platform uses the GitHub API to open a pull request against the main repository on your behalf.

<Steps>
  <Step title="Check the status page">
    Visit `/translations/status` to find which locale needs work. Note the missing or outdated keys so you know what to translate.
  </Step>
  <Step title="Log in with a GitHub-linked WCA account">
    The translation editor requires a WCA account connected to GitHub, because the platform uses the GitHub API (via Octokit) to push changes on your behalf. Go to your account settings to link GitHub if you have not already done so.
  </Step>
  <Step title="Open the translation editor">
    Navigate to `/translations/edit` and select your locale from the dropdown. The editor loads the full content of your locale's YAML file (`config/locales/:locale.yml`).
  </Step>
  <Step title="Edit the YAML content">
    Make your changes directly in the editor. Keep the structure intact — only change values, not key names. Do not remove keys that exist in the English file. You can copy missing keys from the English file (`config/locales/en.yml`) and translate the values.

    Example of a correctly translated key:

    ```yaml
    ja:
      competitions:
        index:
          title: "大会"
    ```
  </Step>
  <Step title="Submit the translation">
    Click **Submit**. The platform:

    1. Strips carriage return characters (`\r`) from the content.
    2. Computes a SHA1 hash of the content to create a unique branch name (`translation-:locale-:hash`).
    3. Syncs your GitHub fork with the upstream `main` branch.
    4. Creates a new branch in your fork pointing to the current upstream HEAD.
    5. Updates `config/locales/:locale.yml` on that branch.
    6. Opens a pull request against `thewca/worldcubeassociation.org` with the commit message `Update :locale translation.`

    The pull request description includes your WCA account details and whether you are a verified translator for this language. Verified translators are members of the WCA Translators group for their language.
  </Step>
  <Step title="Wait for review">
    A WCA staff member or verified translator reviews the pull request. Once merged, the updated translations go live with the next deployment.
  </Step>
</Steps>

## Locale file structure

Each locale file follows the same YAML structure with the locale code as the top-level key:

```yaml
fr:
  activerecord:
    attributes:
      competition:
        name: "Nom"
  competitions:
    index:
      title: "Compétitions"
  common:
    save: "Enregistrer"
    cancel: "Annuler"
  events:
    333: "Cube 3x3x3"
    444: "Cube 4x4x4"
```

The keys mirror the structure used in Rails view helpers (`t('competitions.index.title')`) and in the React components via the `i18n-js` integration. When adding a new translation key to the English source, the corresponding key must be added to all locale files to keep the status page clean.

<Note>
  If you are a fluent speaker of a language not yet listed, you can start a new locale by creating a new YAML file in `config/locales/` and submitting a pull request. The new locale also needs to be added to `lib/static_data/available_locales.json` and registered in the Rails i18n configuration.
</Note>
